## TRAPS — Things That Will Break If You're Not Careful

### 1. DO NOT add ELK `labels` to edges
We tried this (Round 16). Adding `labels: [{width: 50, height: 14}]` to ELK edges forces ELK to reserve space for label boxes, which DOUBLES the graph's vertical size and adds unnecessary bends. The layout becomes stretched and ugly. Use parametric placement instead.

### 2. DO NOT change `base` in vite.config.ts without a strategy
Currently `base: '/ccc-ima-sim/'` for GitHub Pages. Vercel needs `base: '/'`. Changing it breaks GitHub Pages. Use env-var detection: `base: process.env.VERCEL ? '/' : '/ccc-ima-sim/'`.

### 3. DO NOT make MC chart sections collapsible
User explicitly removed this in Round 17. "If you're in the Evidence section, you want to see the data — collapsing adds clicks for zero benefit."

### 4. DO NOT show edge labels on all edges by default
Was tried in multiple rounds. Creates massive visual clutter. The hover-only system is intentional. Only ratio > 0.6 edges show a subtle count.

### 5. DO NOT increase ELK spacing values
Round 18 tuned to `nodeNodeBetweenLayers: 55`. Previous values of 70, 80, 120 all produced too much whitespace. The current values were arrived at through user screenshot feedback.

### 6. The enriched event log is INTENTIONALLY synthetic
The `splitDuration()` function creates fabricated sub-activity durations by dividing real simulation phase timing proportionally. This is by design — it creates richer process mining visualization. The narrative text honestly frames it as "decomposing the 5-step model by case type." DO NOT try to make it more "accurate" or remove it.

### 7. The "Analysis" tab should be ONE merged view
User wanted Process Explorer + Variant Explorer + Throughput Time + Activity Search merged into a SINGLE analysis workspace with modes/panels — like Celonis's Process Explorer with sidebar tabs. Currently it just points to the old `/process-explorer` route. The proper implementation would have the graph centered with switchable sidebar panels (Activities, Variants, Throughput, Search).

### 8. React Flow's markerEnd is architecturally broken for our use
Arrow markers orient based on Handle position (centered on node), NOT the actual SVG path's final segment direction. This means arrows can point into text or at wrong angles. The only fix is either: (a) custom SVG `<marker>` with `orient="auto"` drawn as part of the path, or (b) replace React Flow with pure SVG (recommended).

### 9. The Web Worker matters for Vercel
`src/workers/simulation-worker.ts` runs MC simulations. Vite handles workers with `new Worker(new URL(...), {type: 'module'})`. Verify this works on Vercel's build (it should — Vite bundles workers into the dist).

### 10. The user is a PhD computational scientist
Chi Feng authored the original SSFRC 2017 paper. He cares about: analytical rigor (no fabricated statistics), visual quality (Celonis-level or better), honest framing of synthetic data, and professional presentation. "No such thing as diminishing returns" — keep grinding on quality.

## React Flow → Pure SVG Research Brief

### Why replace
| Problem | Root Cause |
|---------|-----------|
| Arrows point into text | markerEnd orients to Handle, not path end |
| Labels in CSS overlay | Disconnected from SVG coordinate space |
| Single centered Handle per side | Can't use ELK's multi-port routing |
| DOM overhead | Draggable/connectable wrappers we don't use |
| 2.4MB bundle contribution | React Flow + dependencies |

### What pure SVG looks like
```tsx
function SvgProcessGraph({ elkResult }: { elkResult: ElkNode }) {
  const [transform, setTransform] = useState({ x: 0, y: 0, k: 1 });
  // Pan: onMouseDown/Move/Up. Zoom: onWheel.
  return (
    <svg viewBox="..." onWheel={handleZoom}>
      <g transform={`translate(${transform.x},${transform.y}) scale(${transform.k})`}>
        {/* Edges FIRST (behind nodes) */}
        {edges.map(e => <path d={elkPointsToRoundedPath(e.points)} />)}
        {/* Nodes */}
        {nodes.map(n => (
          <g transform={`translate(${n.x}, ${n.y})`}>
            <rect rx={8} width={n.w} height={n.h} fill="white" stroke="#e5e7eb" />
            <circle cx={16} cy={n.h/2} r={10} fill={phaseColor} />
            <text x={32} y={n.h/2 - 2}>{n.label}</text>
            <text x={32} y={n.h/2 + 12} class="freq">{n.freq}</text>
          </g>
        ))}
        {/* Labels on straight segments, offset perpendicular */}
        {edgeLabels.map(l => <text x={l.x} y={l.y}>{l.text}</text>)}
      </g>
    </svg>
  );
}
```

### Libraries to evaluate
- **Pure SVG** — zero deps, ~200 LOC renderer, full pixel control
- **d3-zoom** — mature pan/zoom with inertia, works with any SVG
- **Cytoscape.js** — canvas-based, ELK extension, built-in everything but different paradigm

### Key research question
Is the SVG rewrite worth the effort vs. fixing React Flow incrementally? Prototype FIRST (pure SVG, no interaction, just render), compare visual quality, then decide.
